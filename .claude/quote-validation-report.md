# Quote Validation Report

Validation of all 10 proverb quotes in the Engineering Book of Proverbs application.
Each quote is assessed for accuracy of wording, attribution, and source.

## Quote #1 (001.md)

- **Quote:** "The best code is no code at all."
- **Claimed Author:** Jeff Atwood
- **Claimed Source:** Coding Horror
- **VERIFIED Source:** Blog post "The Best Code is No Code At All" on Coding Horror (blog.codinghorror.com), published May 30, 2007. Later collected in *Effective Programming: More Than Writing Code*.
- **Accuracy:** CORRECT. The quote, author, and source are all accurate. Jeff Atwood wrote this in his Coding Horror blog post where he argued that every new line of code is a liability.
- **Corrections Needed:** None. The source "Coding Horror" is correct, though adding the year (2007) would be a nice addition.

## Quote #2 (002.md)

- **Quote:** "Premature optimization is the root of all evil."
- **Claimed Author:** Donald Knuth
- **Claimed Source:** "The Art of Computer Programming"
- **VERIFIED Source:** "Structured Programming with go to Statements," ACM Computing Surveys, Vol. 6, No. 4, December 1974, pp. 261-301.
- **Accuracy:** ATTRIBUTION CORRECT, SOURCE INCORRECT. The quote is correctly attributed to Donald Knuth, but the source is wrong. It does NOT come from *The Art of Computer Programming*. It comes from Knuth's 1974 paper "Structured Programming with go to Statements" published in ACM Computing Surveys.
- **Additional Context:** The full quote is: "We should forget about small efficiencies, say about 97% of the time: premature optimization is the root of all evil. Yet we should not pass up our opportunities in that critical 3%." Knuth himself later misattributed this to Tony Hoare in his 1989 paper "The Errors of TeX," but research has confirmed Knuth is the original author. Hoare himself stated he did not remember saying it.
- **Corrections Needed:** Change source from "The Art of Computer Programming" to "Structured Programming with go to Statements" (1974).

## Quote #7 (007.md)

- **Quote:** "Make it work, make it right, make it fast."
- **Claimed Author:** Kent Beck
- **Claimed Source:** (none listed)
- **VERIFIED Source:** The concept predates Kent Beck. The earliest known formulation is from Stephen C. Johnson and Brian W. Kernighan in "The C Language and Models for Systems Programming" (Byte magazine, August 1983): "the strategy is definitely: first make it work, then make it right, and, finally, make it fast." Butler Lampson's "Hints for Computer System Design" (1983) contains a similar idea. Kent Beck's related formulation appears in *Test Driven Development: By Example* (2002): "Write a test, make it run, make it right."
- **Accuracy:** PARTIALLY CORRECT. The attribution to Kent Beck is the most common modern attribution and is not entirely wrong, as Beck popularized a version of this philosophy. However, the exact phrasing "Make it work, make it right, make it fast" predates Beck and originates in Unix programming culture (1983).
- **Corrections Needed:** Consider adding a note that this is a Unix programming proverb popularized by Kent Beck, or attributing it to the earlier Johnson/Kernighan source.

## Quote #9 (009.md)

- **Quote:** "The most dangerous phrase in the language is 'We've always done it this way.'"
- **Claimed Author:** Grace Hopper
- **Claimed Source:** (none listed)
- **VERIFIED Source:** The earliest documented version appears in "The Wit and Wisdom of Grace Hopper," published in The OCLC Newsletter, No. 167, March/April 1987. In that source, the wording is "the most damaging phrase in the language is 'We've always done it this way.'" Hopper was known for repeating this sentiment in many talks and speeches.
- **Accuracy:** MOSTLY CORRECT. Grace Hopper is well-documented as having said variants of this quote in speeches and interviews. The attribution is solid. However, the exact wording differs slightly from documented sources:
  - App version: "The most **dangerous** phrase..."
  - Documented version (OCLC 1987): "The most **damaging** phrase..."
- **Corrections Needed:** Consider changing "dangerous" to "damaging" to match the documented source from the 1987 OCLC Newsletter. Both versions circulate widely, but "damaging" appears to be the more historically accurate wording.

## Quote #10 (010.md)

- **Quote:** "Debugging is twice as hard as writing the code in the first place. Therefore, if you write the code as cleverly as possible, you are, by definition, not smart enough to debug it."
- **Claimed Author:** Brian Kernighan
- **Claimed Source:** (none listed)
- **VERIFIED Source:** *The Elements of Programming Style*, 2nd edition, Chapter 2 (1978). Co-authored by Brian W. Kernighan and P. J. Plauger. Published by McGraw-Hill.
- **Accuracy:** CORRECT. The quote and attribution are accurate. The original wording in the book is: "Everyone knows that debugging is twice as hard as writing a program in the first place. So if you're as clever as you can be when you write it, how will you ever debug it?" The version in the app is a widely-used paraphrase that conveys the same meaning but is not word-for-word.
- **Note:** The app version uses "Therefore, if you write the code as cleverly as possible, you are, by definition, not smart enough to debug it" while the original reads "So if you're as clever as you can be when you write it, how will you ever debug it?" Both convey the same idea, but the original is phrased as a question.
- **Corrections Needed:** Consider using the original wording for accuracy, or noting that the version used is a common paraphrase. Also consider adding source: "The Elements of Programming Style" (1978).
